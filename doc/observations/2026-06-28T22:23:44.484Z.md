# Observation: Fix the IPv6 loopback `case` pattern — `[::1]` is a glob class, not a literal

- **Source commit**: `a6a3f549ceb85a78443956007b88a0f3e1197521`
- **Observed at**: `2026-06-28T22:23:44.484Z`
- **Severity**: Medium
- **Category**: Bug
- **File**: `convert_all_gemini_notes.sh:36`
- **Confidence**: 90

## Problem

In the base-URL `case` (line 36), the patterns `http://[::1]` and
`http://[::1]:*` use `[::1]`, which the shell parses as a glob **bracket
character class** (matching a single character from the set `{:, 1}`), not the
literal IPv6 string `[::1]`. As a result the branch never matches a real
`http://[::1]:8317` URL. (ShellCheck SC2102 hints at exactly this.) The
`localhost` and `127.0.0.1` branches are fine.

## Impact

If `CLAUDE_BASE_URL` points at the IPv6 loopback and `CLAUDE_API_KEY` is unset,
the local auto-auth fallback (`export CLAUDE_API_KEY="…local-endpoint"`,
lines 37) is skipped, so the request goes out with no key and likely fails
authentication — even though it is a legitimate local endpoint.

## Suggested Fix

Escape the brackets so they are matched literally:

```bash
case "$base_url" in
    http://localhost|http://localhost:*|http://127.0.0.1|http://127.0.0.1:*|http://\[::1\]|http://\[::1\]:*)
        export CLAUDE_API_KEY="${GEMINI_TO_ORG_LOCAL_AUTH:-local-endpoint}"
        ;;
esac
```

(Alternatively, normalize/extract the host before matching.)

## Verification

Set `CLAUDE_BASE_URL=http://[::1]:8317` with `CLAUDE_API_KEY` unset and confirm
the loopback auth fallback now triggers (the placeholder key is exported).
