# Observation: Guard model-response indexing against an empty `content` list

- **Source commit**: `a6a3f549ceb85a78443956007b88a0f3e1197521`
- **Observed at**: `2026-06-28T22:23:44.482Z`
- **Severity**: Medium
- **Category**: Bug
- **File**: `gemini_to_org.py:909`
- **Confidence**: 80

## Problem

`message.content[0].text` (line 909, and the analogous call at line 473)
assumes the model response has at least one content block. A local
Claude-compatible endpoint that returns `content: []` — a stop with no text, or
an error-shaped body still returned with HTTP 200 — raises `IndexError`, which is
**not** the `ValueError` that the surrounding retry / fail-closed logic catches.

## Impact

A model hiccup crashes the run with a bare `IndexError: list index out of range`
(top-level handler prints it, exit 1) instead of the intended fail-closed retry
and diagnostics. In `_call_shortening_model` (line 473) it aborts the entire
conversion. The Python reviewer reproduced this with a stub client. This directly
undercuts the stated "fail-closed" design of the LLM paths.

## Suggested Fix

Guard both call sites before indexing:

```python
if not message.content:
    raise ValueError("model returned no content")
return message.content[0].text.strip()
# or, tolerant of multi-block responses:
# return "".join(b.text for b in message.content if getattr(b, "text", None)).strip()
```

## Verification

Add a `FakeClient`-backed test returning `content=[]` and assert the failure
surfaces through the existing `ValueError` retry/abort path rather than as an
`IndexError`. Note the current tests always monkeypatch `_call_model`, so this
real response-parsing path is presently uncovered.
