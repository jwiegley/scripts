# Observation: Stop truncating the transcript on a bare "Transcription ended after" line

- **Source commit**: `a6a3f549ceb85a78443956007b88a0f3e1197521`
- **Observed at**: `2026-06-28T22:23:44.474Z`
- **Severity**: High
- **Category**: Bug
- **File**: `gemini_to_org.py:2456`
- **Confidence**: 85

## Problem

The end-of-transcript detection (around lines 2456-2464) builds `end_candidate`
by stripping leading `#`/`**` from *any* line — not only Markdown headings — and
`break`s out of the transcript parse loop on the first line matching
`^Transcription ended after\s+.+$`. A normal spoken/continuation line that
happens to begin with that phrase therefore terminates parsing.

## Impact

Every transcript line after such a line is silently dropped from the Org output.
For a meeting-notes converter, silent content loss is the worst possible failure
mode — the user has no signal that anything is missing. The Python reviewer
verified this empirically: a body line such as

> "Transcription ended after we finished, but actually we kept going."

causes a subsequent `00:05:00` timestamp section to be omitted entirely from the
output.

## Suggested Fix

Only treat the line as the end marker when the *source* line is genuinely a
heading, and/or anchor on the timestamped form Google actually emits (the real
marker is followed by a duration, not arbitrary prose):

```python
# require the line to parse as a heading first, OR tighten the regex:
if heading is not None and re.match(r'^Transcription ended after\s+\d', end_candidate):
    break
```

## Verification

Add a regression test that feeds a transcript whose *body* contains a non-heading
line beginning "Transcription ended after …" followed by more timestamped
content, and assert the later content survives in the converted output. Then run
`python test_gemini_to_org.py`.
