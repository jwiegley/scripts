# Observation: Do not let a duplicate normalized section heading discard content

- **Source commit**: `a6a3f549ceb85a78443956007b88a0f3e1197521`
- **Observed at**: `2026-06-28T22:23:44.481Z`
- **Severity**: High
- **Category**: Bug
- **File**: `gemini_to_org.py:1367`
- **Confidence**: 70

## Problem

`_store_section` (lines 1367-1380) replaces an already-stored section whenever
the new block has greater-or-equal non-blank line count
(`len(nonblank) >= len(existing_nonblank)`). If a transcript legitimately
contains two headings that normalize to the same section name — e.g. two
"Summary"/"Details" blocks, or the meeting-title-with-"Transcript" heuristic
colliding with a real "Transcript" heading — one block is silently discarded
rather than the two being combined.

## Impact

Whole-section content loss. The Python reviewer verified that with two "Summary"
blocks, the longer one replaces the shorter (dropping the first); when the second
is shorter, it is instead discarded. Either way one block's content disappears
from the output with no warning.

## Suggested Fix

Concatenate/merge sections that normalize to the same name, or keep the first and
emit a stderr warning, rather than length-based replacement. If de-duplication is
genuinely intended for some sections, log the discarded block so the loss is at
least visible:

```python
if name in self.sections:
    # merge instead of replace, or:
    print(f"warning: duplicate section '{name}' — merging", file=sys.stderr)
```

## Verification

Add a test with two headings that normalize to the same section name (one case
where the second block is longer, one where it is shorter) and assert that no
content is lost — either both blocks are present (merged) or a warning is
emitted.
