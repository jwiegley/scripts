# Observation: Constrain the model endpoint to loopback to prevent transcript exfiltration

- **Source commit**: `a6a3f549ceb85a78443956007b88a0f3e1197521`
- **Observed at**: `2026-06-28T22:23:44.477Z`
- **Severity**: High
- **Category**: Security
- **File**: `gemini_to_org.py:115`
- **Confidence**: 90

## Problem

`validate_claude_base_url` (lines 115-131) rejects only `anthropic.com` /
`*.anthropic.com`; every other host is accepted. The default base URL is
`http://localhost:8317`, but `CLAUDE_BASE_URL` / `--base-url` can repoint the
client at any host. Transcript task inference is **enabled by default** (see
`convert_all_gemini_notes.sh:28`), and `_call_model` embeds the *entire*
transcript into the prompt body (`comparison_prompt`, lines 934-975) sent via
`client.messages.create` (lines 904-907).

## Impact

The tool is named and documented as using a *local* model, but a wrong or
attacker-influenced `CLAUDE_BASE_URL` silently exfiltrates entire confidential
meeting transcripts (plus inferred action items) to an arbitrary network
endpoint — potentially over plain HTTP. Because the only guard is an
Anthropic-domain denylist, it gives false assurance that egress is constrained
to the local machine. A single mis-set environment variable (easily copied from
another tool) leaks every meeting's content with no warning.

## Suggested Fix

Make the check an allowlist rather than a single-domain denylist: by default
require the host to resolve to loopback (`localhost` / `127.0.0.1` / `::1`), and
require an explicit opt-in flag (e.g. `--allow-remote-endpoint`) to permit any
other host. At minimum, print the resolved endpoint host on stderr at startup so
unexpected egress is visible even in non-verbose runs.

## Verification

Add tests asserting that a non-loopback `base_url` is rejected without the
opt-in flag and accepted with it. Manually run with
`CLAUDE_BASE_URL=http://example.com` and confirm the tool refuses (or loudly
warns) before sending any transcript.
