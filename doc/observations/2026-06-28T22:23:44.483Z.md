- **File**: `gemini_to_org.py:375`
- **Confidence**: 75

## Problem

`ParticipantDatabase.load()` / `save()` (lines 375-384) open the JSON database
*without* `encoding='utf-8'` (so the encoding is locale-dependent), and `save()`
truncates-and-writes in place (non-atomic). `load()` also does not validate that
the parsed JSON is a flat `str -> str` dict before `get_id` (lines 386-390)
indexes it. The rest of the codebase is otherwise consistently explicit about
`encoding='utf-8'`.

## Impact

This database persists stable participant UUIDs across *all* past notes, so
corruption has broad blast radius. Specifically:

- Non-ASCII participant names can be mangled, or raise `UnicodeEncodeError`, on a
  non-UTF-8 locale.
- An interrupted `save()` corrupts the DB and loses stable IDs everywhere.
- A non-dict parse (corrupt or hand-edited `--db`) causes an uncaught
  `TypeError` at use time; a pathologically large array could exhaust memory.

The existing `test_corrupt_participant_database_fails_loudly` shows a corrupt DB
already aborts conversion, which amplifies the cost of a bad write.

## Suggested Fix

```python
# load:
with open(self.path, encoding='utf-8') as f:
    data = json.load(f)
if not isinstance(data, dict):
    raise ValueError(f"participant DB {self.path} is not a JSON object")
self.participants = data

# save (atomic, same dir):
import os, tempfile
fd, tmp = tempfile.mkstemp(dir=os.path.dirname(self.path) or '.', suffix='.tmp')
with os.fdopen(fd, 'w', encoding='utf-8') as f:
    json.dump(self.participants, f, indent=2, sort_keys=True, ensure_ascii=False)
os.replace(tmp, self.path)
```

Consider `0o600` permissions on the new file.

## Verification

Add tests: round-trip a non-ASCII participant name; a non-dict DB file raises a
clear error rather than `TypeError`; an interrupted save (simulated) leaves the
prior DB intact.
