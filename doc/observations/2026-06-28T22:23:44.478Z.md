# Observation: Add `set -euo pipefail` and make the loop counters safe under it

- **Source commit**: `a6a3f549ceb85a78443956007b88a0f3e1197521`
- **Observed at**: `2026-06-28T22:23:44.478Z`
- **Severity**: High
- **Category**: Bug
- **File**: `convert_all_gemini_notes.sh:1`
- **Confidence**: 92

## Problem

Two coupled issues:

1. The wrapper has no `set -euo pipefail`. The repository convention (per
   `CLAUDE.md`) is `#!/usr/bin/env bash` with `set -euo pipefail`, and there is
   no comment explaining the omission.
2. The loop counters use the post-increment form `((converted++))`,
   `((skipped++))`, `((errors++))` (lines 62, 65, 72). The exit status of
   `((var++))` reflects the *pre-increment* value, so when a counter is `0` the
   expression evaluates to `0` and returns **exit status 1**.

These interact: the obvious fix for (1) — adding `set -e` — turns (2) into a
script-aborting bug.

## Impact

As written today (no `set -e`), unset-variable typos expand to empty silently
and intermediate command failures do not halt the batch. If a maintainer later
adds `set -euo pipefail` to follow the convention *without* touching the
counters, the very first `((converted++))` (or first skip/error) exits the script
mid-loop under `set -e`, silently truncating the batch after a single file.

## Suggested Fix

Apply both changes together:

```bash
#!/usr/bin/env bash
set -euo pipefail
...
((++converted))      # pre-increment is always status 0
# or: converted=$((converted + 1))
# or: : $((converted++))
```

convertible, skipped (non-Gemini), and failing files; confirm all files are
processed and the printed `Converted/Skipped/Errors` summary is correct.
